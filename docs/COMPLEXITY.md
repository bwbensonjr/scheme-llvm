# Code Complexity Catalogue

A reproducible catalogue of the code in this repository, classified by **component**,
**role**, and **language**. It exists to answer one question honestly: *how much code is
there, and how much of it did we actually write by hand?*

A raw line-of-code total is misleading here. Most of the tree is **not** hand-authored
system code — it is generated LLVM IR (the committed compiler bootstrap snapshots),
vendored third-party code, an imported language-standard reference, and OpenSpec tracking
history. The catalogue below separates those roles so the **authored subtotal** — the
compiler, runtime, standard library, and tooling we maintain — stands apart from the
full-tree total.

> **LOC is a coarse proxy for complexity.** Line counts measure *volume*, not cognitive or
> cyclomatic difficulty. Use this catalogue to see *where* the code is and *what kind* it
> is, not to rank difficulty. The role and component split is the deliverable; the headline
> number is not.

## Roles

Every tracked file is assigned exactly one role:

| Role | Meaning |
|---|---|
| **authored** | Hand-written compiler / runtime / stdlib / build + dev tooling source — the headline subtotal. |
| **docs** | Hand-written project prose (README, `docs/`, design notes). |
| **generated** | Build artifacts checked into the tree (`bootstrap/*.ll`). |
| **vendored** | Third-party code copied into the tree (none currently — the rule remains for any future copy). |
| **reference** | External material reproduced for convenience (imported specs, genesis). |
| **tracking** | OpenSpec changes + specs — process history, not system code. |
| **config** | Agent / editor / repo configuration (`.claude`, `.gitignore`). |
| **test** | Test programs and harnesses. |
| **demo** | Example programs under `demos/`. |
| **spike** | First-party experiments (none currently — the rule remains for any future spike). |
| **other** | Anything the rules do not match — a visible gap to fix, never dropped. |

## Regenerating

The tables below are produced by [`tools/complexity.sh`](../tools/complexity.sh) and
regenerated with:

```sh
make catalogue
```

When code moves, the numbers below go stale until you regenerate. If a file lands in the
wrong component or role — or in `other` — fix the **classification policy** (the single
ordered `case` block in `tools/complexity.sh`), then rerun `make catalogue`. That script is
the one place the authored-vs-generated-vs-vendored policy lives.

Each regeneration also appends the current totals to [`complexity-history.csv`](complexity-history.csv)
— an append-only, long-format ledger (`timestamp,commit,dimension,key,files,loc`) of the
whole-tree total, each role, and each component over time. A snapshot is recorded only when
the numbers change from the last one, so the history tracks real movement rather than every
run. The history file is itself excluded from the count (a metric that included its own
ledger could never settle).

## Reading the numbers

<!-- Hand-authored interpretation. Refreshed alongside the tables (the complexity-catalogue
     skill updates this section); it lives OUTSIDE the generated markers so regenerating the
     tables never overwrites it. -->

- **The system we maintain is small.** The authored subtotal is a low-single-digit
  percentage of the full tree; the rest is generated IR, imported reference, and OpenSpec
  tracking. Complexity work should focus on the authored components.
- **`bootstrap/*.ll` dominates the raw total** and is generated — it is the compiler
  compiled to LLVM IR, checked in as a stage-0 bootstrap input, not code anyone edits.
- **Authored code spans three languages, and the heaviest files sit in each one:** the C
  runtime (`src/runtime/runtime.c`, ~1.2K), the Scheme codegen (`src/emit.ss`, ~1.0K), and
  the C++ native driver (`src/emit.cpp`, ~0.9K). `compiler-core` (Scheme, ~4.2K over 11
  files — codegen, the Chez-hosted `compile.ss` driver, the parser, the REPL core, the
  prelude) is the largest authored component; `runtime` (C, ~1.2K) and `native-drivers`
  (C++, ~1.1K — `emit.cpp` plus `run-boot.cpp`) are each roughly a thousand lines. The
  frontend pass ladder (`src/passes/`, ~0.6K) is deliberately lean.
- **OpenSpec tracking is the second-largest mass** and larger than the authored system —
  expected for a spec-driven project, and mostly archived change history.
- **No vendored third-party code remains.** The nanopass copy and the first-party spikes
  were retired once the front-end decisions settled (change `retire-spikes`); their findings
  live in the specs and archived changes. This cut ~20K LOC (~8.7%) from the tree.
